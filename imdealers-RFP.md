# 아임딜러 플랫폼 구축 RFP

> **[AI Agent 필독]**
> 이 문서는 개발 에이전트가 프로젝트 전반을 이해하기 위한 요구사항 문서다.
> 세부 기능보다 **서비스 철학과 브랜드 방향**을 먼저 이해하고 모든 구현에 반영해야 한다.
> 코드 한 줄 작성 전에 이 문서를 완전히 숙지할 것.

---

## 핵심 키워드

`AI 기반 진짜견적` · `신뢰 구조` · `빠른 베타 오픈` · `고객 리텐션` · `SaaS 확장`

---

## 3단계 로드맵 전체 구조

| 단계      | 핵심 목표   | 집중 포인트                                        |
| --------- | ----------- | -------------------------------------------------- |
| **1단계** | 시장 진입   | 프론트 경험, AI 인상, 신뢰 구조, 빠른 베타 오픈    |
| **2단계** | 고객 리텐션 | 보험·사고·운행일지·담당자 연결 기반 고객 접점 유지 |
| **3단계** | 플랫폼 확장 | 내부 운영 체계 표준화 및 외부 SaaS 확장            |

> **현재 개발 범위: 1단계 집중** — 2·3단계는 확장 구조 고려를 위한 참고용

## 1. 프로젝트 개요

아임딜러는 기존 자동차 장기렌트·리스 시장에서 반복되어 온 허위견적, 낚시견적, 상담접수형 DB 수집 구조와는 다른 방향의 서비스를 구축하고자 한다.

본 프로젝트의 핵심은 **단순한 견적 비교 웹사이트 제작이 아니다.** 고객이 먼저 탐색하고, 먼저 이해하고, 먼저 신뢰할 수 있는 AI 기반 진짜견적 서비스를 만드는 것이 출발점이다.

---

## 2. 프로젝트가 지향하는 방향

기존 시장의 문제:

- 정확한 견적보다 상담 유도와 DB 수집이 우선
- 고객은 비교하는 것이 아니라 영업을 당한다고 느낌

아임딜러가 뒤집고자 하는 것:

- 고객이 먼저 AI 기반으로 차량과 조건을 탐색
- 실제 운영 가능한 진짜견적을 확인
- 상담은 그 이후 자연스럽게 대화로 연결

**4가지 핵심 방향:**

1. 견적을 미끼로 상담을 유도하는 방식이 아닌, 고객이 먼저 판단할 수 있는 구조
2. AI가 전면에 보이되 과장되지 않고 실제 도움을 주는 구조
3. 계약 전환 이후에도 고객과 관계를 유지하는 구조
4. 내부에서 검증한 운영 체계를 바탕으로 외부 SaaS 확장까지 고려한 구조

---

## 3. 단계별 스토리라인

### 1단계 — 시장에 먼저 안착하는 단계

목적: 전 차종을 완벽하게 구현하는 것이 아니라, 고객이 처음 접하는 순간 **"이 서비스는 기존 견적사이트와 다르다"** 는 인식을 갖게 만드는 것

- AI 기반 추천 경험이 전면에 보이는 프론트엔드 웹
- 핵심 인기 차종 중심의 견적 시스템
- 실제 운영이 가능한 관리자 페이지
- 빠른 베타 오픈 / 차별화된 첫인상 / 신뢰감 있는 탐색 구조

### 2단계 — 고객을 붙잡는 단계

계약 이후 고객과의 관계를 끊지 않는 단계

- 보험, 사고, 운행일지, 영업담당자 연결 → 고객 반복 접점
- 재계약, 추가계약, 소개를 자연스럽게 유도하는 구조
- 고객 생애가치(LTV)를 높이는 구조
- 아임딜러를 계약 이후까지 관리하는 파트너로 인식

### 3단계 — 플랫폼으로 확장하는 단계

내부에서 검증된 운영 방식을 외부 소형 에이전시로 확장

- 내부 영업 중심 회사 → 운영 시스템 제공 회사로 전환
- 소형 에이전시가 활용할 수 있는 SaaS형 CRM 확장
- 상생 모델 기반 플랫폼 사업자 구조 구축

## 4. 1단계 구축 방향

### 4-1. 목표

1단계 목표 = 전 차종 완성형 플랫폼 구축 **아님**

핵심: AI 기반 진짜견적 서비스를 빠르게 시장에 선보이고, 고객에게 기존 사이트와 완전히 다른 경험 제공

- 프론트 개발을 최우선으로 한다
- AI 경험을 반드시 전면에 배치한다
- 전 차종이 아닌 핵심 인기 차종 중심으로 시작한다
- 상담접수형 구조는 배제한다

### 4-2. 프론트엔드 방향

- 첫 화면에서 기존 견적사이트와 다르다는 인상을 줄 것
- AI 기반 탐색 서비스처럼 보일 것
- 데스크톱 웹 레이아웃을 우선 기준으로 개발하되, 반응형으로 모바일도 지원할 것
- 상담 유도보다 탐색과 이해가 앞서는 구조일 것

### 4-3. AI 경험

> AI는 처음부터 완성형일 필요 없음. "걸음마 단계"임을 솔직하게 보여주는 것 자체가 브랜드 신뢰에 도움

- 업종, 사용 목적, 예산, 성향 기반 추천
- 추천 차량 및 추천 조건 제안
- 추천 이유의 짧고 명확한 설명
- 향후 AI 고도화를 위한 데이터 축적 구조

### 4-4. 견적 시스템

핵심 차종 약 20개 내외 MVP로 시작

견적 변수:

- 차량가 / 계약기간 / 약정거리 / 보증금·선납금
- 잔존가치 / 금리 / 수수료 / 세금 및 부대비용
- 프로모션 적용 여부

운영 원칙:

- 실제 운영 가능한 견적 구조
- 사람이 보완 가능한 운영 구조 병행

### 4-5. 관리자 페이지

완벽한 자동화보다 **운영자가 즉시 수정·보완할 수 있는 실전 도구**

- 차량 및 트림 등록/수정/비노출
- 견적 운영값 관리
- 추천 문구 및 AI 기초 데이터 관리
- 메인 화면 및 콘텐츠 운영
- 상담 흐름 확인 및 운영 메모 기록

### 4-6. 디자인 방향

디자인 목표 = 예쁜 웹사이트가 아님

고객에게 줘야 할 인상:

- "지금까지 없던 서비스"
- "AI 기반 서비스"
- "신뢰가 가는 서비스"

구현 방향:

- 기존 허위견적 사이트와 다른 분위기
- 기술 기반 서비스처럼 느껴지는 화면
- 과장되지 않은 AI 인상
- 상담 압박이 없는 구조
- 정보가 정돈되어 있고 투명해 보이는 느낌

### 4-7. 상담 구조

채널톡 중심 운영 / AI형 대화 흐름으로 설계

```
❌ 절대 금지
- 이름/전화번호를 먼저 입력해야 견적을 보여주는 방식
- 견적 보기 전에 상담신청을 강요하는 방식
- DB 수집이 주 목적처럼 보이는 화면 구성
- 단순 '상담원 연결' 반복형 챗봇

✅ 반드시 구현
- 차량 조건, 사용 목적, 예산, 성향을 이어받는 AI형 상담 흐름
```

## 5. 1단계 상세 요구사항

### 5-1. 고객용 프론트엔드 웹

- 반응형 웹 기반 구축
- 데스크톱 웹 레이아웃 우선 개발, 모바일은 반응형으로 지원
- AI 추천 진입 구조 명확화
- 핵심 차종 탐색 구조 제공
- 개인정보 입력 전에도 서비스 핵심을 경험할 수 있는 구조
- 브랜드 신뢰 요소가 화면 안에 자연스럽게 배치

### 5-2. AI 추천 경험

AI는 화면 한 구석 기능이 아니라 **서비스의 중심 경험**

입력 변수:

- 업종 또는 사업 형태
- 차량 사용 목적
- 예산 범위 및 납입 성향
- 주행거리 및 인수/반납 성향

출력:

- 추천 차량 1~3개 제안
- 추천 이유 설명 (짧고 명확하게)
- 조건별 비교 관점 제시

### 5-3. 견적 결과 구조

"월 xx원부터"만 강조하는 방식 금지 → 고객이 실제로 비교·판단할 수 있게 구성

필수 표시 항목:

- 차량명 및 트림
- 월 납입금
- 계약기간 및 연간 약정거리
- 보증금/선납금 여부
- 인수형/반납형 구분
- 추천 배지 및 AI 해설
- 비용처리 또는 운영 관련 체크포인트
- 채널톡 연계 버튼

옵션 (가능하면):

- 보수형 / 표준형 / 공격형 3개 시나리오 제시

### 5-4. 견적 시스템 로직

- 핵심 차종 20개 내외 우선 적용
- 차량가, 기간, 거리, 금리, 잔존가치, 수수료 등 기본 변수 반영
- 실제 계약 가능한 수준의 견적 기준 적용
- 유인용 최저가 방식 금지
- 운영자가 빠르게 수정 가능한 구조
- 특정 차종/조건의 노출 제어 가능
- 오차 발생 시 운영 보정이 가능한 구조

### 5-5. 관리자 페이지 (Admin)

- 차량 및 트림 관리 (등록/수정/삭제/비노출)
- 견적 변수 관리 (금리, 잔존가치 등)
- 차종 노출/비노출 관리
- 추천 문구 및 AI 기초 데이터 관리
- 메인 화면 및 콘텐츠 운영
- 채널톡 연결 흐름 확인
- 운영 메모 기록
- 기본 권한 구분 (관리자 등급)

### 5-6. 데이터 및 로그 구조

1단계부터 AI 고도화를 위한 데이터 축적 구조 필수 설계

수집 항목:

- 고객 탐색 흐름 기록
- 추천 선택 로그 저장
- 차종 선호 데이터 축적
- 조건별 이탈 지점 파악
- 채널톡 연계 로그 축적

목적: 향후 AI 추천 고도화에 활용 가능한 데이터 자산 확보

### 5-7. 운영 원칙

- 고객 경험은 끊기지 않게 설계할 것
- 시스템이 부족한 부분은 사람이 백업할 수 있을 것
- 고객에게 혼란을 주지 않도록 커뮤니케이션 문구를 설계할 것
- 솔직함을 신뢰로 연결할 것
- AI·견적·상담 흐름이 하나의 브랜드 경험으로 이어질 것

## 6. 1단계 성공 기준

| 기준            | 판단 질문                                                   |
| --------------- | ----------------------------------------------------------- |
| 차별성          | 고객이 기존 견적사이트와 다르다고 느끼는가                  |
| AI 인상         | AI 기반 서비스라는 인상이 분명한가                          |
| 신뢰 구조       | 허위견적·상담접수형 사이트처럼 보이지 않는가                |
| 운영 가능성     | 인기 차종 기준으로 실제 운영이 가능한가                     |
| 백오피스 보완력 | 운영자가 관리자 페이지에서 서비스를 충분히 보완할 수 있는가 |

## 7. AI Agent가 반드시 기억해야 할 핵심 원칙

```
1. 단순 자동차 견적 사이트가 아니다
2. 1단계는 완벽한 자동화보다 빠른 시장 진입이 우선이다
3. 프론트 경험과 브랜드 인상이 가장 중요하다
4. 상담접수형 구조로 보이면 실패다 ← 절대 원칙
5. AI는 완벽하지 않아도 되지만 존재감은 분명해야 한다
6. 시스템이 부족한 부분은 운영으로 보완하더라도 고객 경험은 끊기면 안 된다
7. 2단계와 3단계 확장을 고려한 구조적 설계가 필요하다
```

> 세부 개발 스코프는 착수 전 상세 협의와 설계 과정에서 조정 가능하나,
> 본 문서에 담긴 **서비스 방향성, 스토리라인, 브랜드 철학, 운영 원칙**은
> 프로젝트 전반에 일관되게 반영되어야 한다.
